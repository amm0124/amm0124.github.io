# 서론

Database - query를 작성 하기 전, 알아야 할 postgresql의 기본 문법 및 개념에 대해 알아보자.

## SQL 

SQL의 약자는 `standard query language`입니다. 즉 정형화된 질의 언어라는 뜻입니다.
이를 가지고 사용자는 database에 요청을 할 수 있습니다. 
SQL의 문법에 대해 알아보겠습니다.

## DDL, DML, DCL, TCL

SQL을 어떻게 쓰는지, 예를 들어 `insert .. ~`를 알아보기 이전에, 위 4가지에 대해 알아봅시다.
일단 마지막이 L로 끝납니다. 이는 Language를 뜻합니다. 항상 새로운 단어를 들어보면 단어의 뜻을 생각해야 합니다. 이를 바탕으로 위 4가지 단어에 대해 파악해봅시다.
DCL과 TCL은 간략하게 다루겠습니다.

- DDL : Data Definition Language (데이터 정의 언어)
- DML : Date Manipulation Language (데이터 조작 언어)
- DCL : Data Control Language (데이터 컨트롤 언어)
- TCL : Transaction Control Language (트랜잭션 컨트롤 언어)

## DDL : Data Definition Language

데이터를 정의하는 언어입니다. 간단하게 생각해서 database table을 정의하는 Language를 `DDL`이라고 합니다.
예시를 들어보면 `Create` 키워드가 있겠네요. 또, `ALTER(수정), DROP, DELETE` 등을 통해서 table을 수정, 삭제하는 것 모두 `DDL` 문법이라고 생각할 수 있습니다.

아래는 database table을 create하는 예시입니다.

    create table classroom (
        building varchar(15),
        room_number varchar(7),
        capacity numeric(4,0),
        primary key (building, room_number)
    );

create 명령어를 통해 classroom이라는 table을 만들었습니다. 내부 column들은 보시는 것처럼 설정했습니다. 
`create`를 통해 table에 들어갈 data를 `제한`했습니다. 이는 바로 data를 정의했다고 볼 수 있습니다. 

`DDL`은 `database table을 정의하는 즉, table에 자체에 작업을 수행하는 문법`이라고 생각하시면 될 것 같습니다.

## DML : Data Manipulation Language

데이터를 조작하는 언어입니다. 데이터를 조작(수정)하기 위해선, DDL을 통해 Data를 정의해야 합니다. 이후 DML 문법을 통해 data를 `조작(조회 하는 것 또한 포함합니다)`합니다. 조작의 종류에는 `update, insert, delete, select`가 있습니다. 
우리는 DDL을 통해 table 및 데이터를 정의하고, DML을 통해 정의된 데이터에 `조작`을 합니다. 

## DCL : Data Control Language, TCL : Transactional Control Language

DCL(Data Control Language)은 데이터를 컨트롤하는 언어입니다. 예를 들어, 어떤 데이터는 특정 `권한`을 지닌 사람만 볼 수 있도록 설정하는 것이 예시입니다. 예시로 `Grant(권한 부여), Revoke(권한 제거)` 등의 문법을 사용합니다. 

TCL(Transaction Control Language)은 트랜잭션을 컨트롤하는 언어입니다. `트랜잭션이란 어떤 여러 작업을 하나의 작업으로 보는 것 `이라고 생각하면 편합니다. 트랜잭션이 성공적으로 완료되었으면 이를 저장하는 `Commit`, 문제가 생기면 취소하는 `Rollback`, 트랜잭션을 되돌리기 위해 중간 savepoint를 정하는 `savepoint` 등의 명령어가 `TCL`입니다.

## Integrity : 데이터베이스의 정확성과 일관성을 보장하기 위해 설정되는 조건

`DBMS(DataBase Management System : 데이터베이스 관리 시스템)`을 사용하면, 어떤 table이 특정 table의 column을 참조하도록 만들 수 있습니다. A table의 A1 column이 B table의 B1이라는 column을 참조하는 상황입니다. B1이 참조하는 A1이 수정되면 B1은 어떻게 되어야 할까요? 이러한 문제가 바로 `Data Integrety(데이터 무결성)` 문제입니다. 무결성 문제에는 3가지가 존재합니다. `primary key, foreign key, not null`이라는 3가지 문제가 있습니다. 차례로 알아봅시다.

## Integrity - key

Table을 생성할 때, 두 가지의 `key`가 존재합니다. `primary key(pk)와 foreign key(fk)`입니다. pk는 `어떤 table을 대표하는 column들의 집합`입니다. pk는 테이블을 대표하는 column들의 쌍이기 때문에, 절대 중복되는 데이터가 존재해선 안 됩니다. 만약 중복되는 pk를 넣으려고 시도하면, 무결성 문제 때문에 `삽입이 거부`됩니다.

`다른 table의 column들을 참고하는 column들을 fk`라고 합니다. 이는 다른 테이블을 참조하는 것이고, 다시 말해서 외부에 의존하는 column들입니다. 만약 외부 테이블의 참조되는 column이 삭제되면 어찌 하나요? 상황에 따라서는 데이터를 가만히 놔두어야 할 때도 있고, null로 설정해야 할 때도 있습니다. 이러한 문제에 관한 것이 `foreign key Integrity`입니다.

어떤 column들은 절대로 NULL이 되어선 안 되는 상황이 존재할 수 있습니다. `이름이라는 column`이 필수적으로 필요한 상황에선, 절대로 이름이란 column을 NULLable해선 안 되겠죠. 이를 `NOT NULL`이라고 합니다.
